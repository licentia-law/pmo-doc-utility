## 2. 제품 개요

### 2.1 제품명(가칭)
DocMate for Proposal  
또는  
Proposal Utility Suite
또는
PMO Portal

최종 프로그램명은 추후 확정한다.

### 2.2 제품 목적
사내 제안서 작성 및 제출 과정에서 반복적으로 수행하는 문서 작업을 한 곳에서 빠르게 처리할 수 있는 Windows 데스크톱 앱을 구축한다.

### 2.3 해결하려는 문제
현재 업무에서는 아래 작업이 자주 반복된다.

- PPTX를 PDF로 변환
- 여러 PDF를 순서대로 병합
- 제출 제한 용량에 맞게 PDF 용량 조절
- 자주 사용하는 업무 사이트로 이동

이 과정에서 여러 프로그램과 사이트를 오가야 하며, 작업 시간이 늘어나고 실수가 발생할 가능성이 높다. 또한 외부 웹사이트 업로드 방식은 보안상 부담이 있다.

### 2.4 제품 목표
- 로컬 환경에서 PPTX를 고품질 PDF로 변환한다.
- 여러 PDF를 순서대로 병합할 수 있다.
- 제출 요구 용량에 맞춰 PDF 용량을 줄일 수 있다.
- 제안서 패키징 플로우를 한 번에 처리할 수 있다.
- 자주 사용하는 업무 사이트를 빠르게 열 수 있다.
- 인터넷 없이도 핵심 문서 변환 기능이 동작한다.
- 사내 20명 내외 인원이 쉽게 설치하고 사용할 수 있다.

## 3. 배경 및 방향성

### 3.1 배경
사내 제안서 작성 시 표지, 목차, 정성, 정량, 부록 등을 각각 별도의 PPTX로 작성하고, 이를 PDF로 변환한 뒤 최종 PDF로 병합하는 작업이 빈번하게 발생한다. 또한 제안요청서에서 PDF 업로드 용량 제한을 두는 경우가 많아, 용량 조절도 실무상 필수 기능이다.

### 3.2 제품 방향
본 제품은 웹 서비스가 아닌 Windows 데스크톱 앱으로 개발한다. 이유는 다음과 같다.

- PPTX to PDF 품질 확보가 중요하다.
- 문서 파일을 외부 서버에 업로드하지 않고 로컬에서 처리하는 것이 보안상 유리하다.
- 인터넷 차단 환경에서도 사용 가능해야 한다.
- Windows + Office 365 + PowerPoint 설치 환경이 이미 확보되어 있다.

### 3.3 핵심 가치
- 빠름
- 단순함
- 로컬 처리
- 품질 우선
- 반복 업무 최소화

## 5. 문제 정의

### 5.1 현재 불편
- PPTX to PDF를 수동으로 반복 수행해야 한다.
- PDF 병합을 위해 별도 프로그램 또는 사이트를 이용해야 한다.
- 용량 제한을 맞추기 위해 여러 번 압축을 시도해야 한다.
- 자주 사용하는 사이트 접근이 분산되어 있다.
- 작업 이력이 남지 않아 반복 작업 추적이 어렵다.

### 5.2 해결 방향
- PPTX to PDF를 로컬 PowerPoint 기반으로 자동화한다.
- PDF 병합과 용량 조절을 앱 내부 기능으로 제공한다.
- 여러 문서를 한 번에 등록하고 처리하는 제안서 패키징 기능을 제공한다.
- 업무용 사이트 즐겨찾기 섹션을 제공한다.
- 작업 기록을 로컬에 남기고 삭제할 수 있게 한다.

## 8. 핵심 사용자 시나리오

### 8.1 시나리오 A: 단일 PPTX를 PDF로 변환
1. 사용자는 홈 화면에서 "PPTX → PDF"를 선택한다.
2. 파일을 선택한다.
3. 저장 경로를 지정한다.
4. 변환을 실행한다.
5. PDF 결과를 저장한다.
6. 필요 시 결과 폴더를 연다.

### 8.2 시나리오 B: 여러 PDF를 병합
1. 사용자는 홈 화면에서 "PDF 병합"을 선택한다.
2. 여러 PDF를 추가한다.
3. 드래그로 순서를 변경한다.
4. 병합을 실행한다.
5. 최종 PDF를 저장한다.

### 8.3 시나리오 C: 제안서 패키징
1. 사용자는 홈 화면에서 "제안서 패키징"을 선택한다.
2. 표지, 목차, 정성, 정량, 부록 파일을 등록한다.
3. PPTX/PDF 혼합 파일 목록을 순서대로 정리한다.
4. 최종 파일명을 자동 제안받는다.
5. PPTX는 PDF로 변환되고, PDF는 유지된다.
6. 전체 문서를 병합한다.
7. 필요 시 최종 용량 조절을 적용한다.
8. 최종 결과 PDF를 저장한다.

### 8.4 시나리오 D: 자주 사용하는 사이트 열기
1. 사용자는 홈 화면 하단의 즐겨찾기 섹션을 본다.
2. 원하는 사이트 카드를 클릭한다.
3. 사이트가 기본 브라우저로 열린다.

### 8.5 시나리오 E: 작업 기록 확인 및 삭제
1. 사용자는 작업 기록 화면으로 이동한다.
2. 작업 시간, 기능 종류, 파일명, 원본 경로를 확인한다.
3. 개별 삭제 또는 전체 삭제를 수행한다.

## 9. 기능 요구사항

## 9.1 기능 1: PPTX → PDF 변환

### 목적
PowerPoint 파일을 고품질 PDF로 로컬 변환한다.

### 입력
- 파일 형식: .pptx

### 출력
- PDF 파일

### 상세 요구사항
- 단일 파일 선택 가능
- 드래그 앤 드롭 가능
- 저장 위치 지정 가능
- 변환 완료 후 결과 폴더 열기 가능
- 변환 성공/실패 메시지 표시
- 기본값은 품질 우선
- 변환 시 인터넷 연결이 필요하지 않아야 함

### 제약/정책
- 1차는 .pptx만 공식 지원
- .ppt는 추후 확장 검토
- PowerPoint 설치 환경을 전제로 함

### 오류 처리
- 파일 열기 실패
- PowerPoint 호출 실패
- 저장 경로 권한 부족
- 파일 잠금
- 문서 손상
- 경로 길이/문자열 문제

### 사용자 안내 문구
- 원본 PPT에 사용된 폰트가 현재 PC에 설치되어 있지 않으면 결과 PDF의 레이아웃이 달라질 수 있습니다.
- 변환 후 최종 PDF를 반드시 확인해 주세요.

---

## 9.2 기능 2: PDF 병합

### 목적
여러 PDF를 하나의 PDF로 합친다.

### 입력
- PDF 여러 개

### 출력

### 상세 요구사항
- 여러 PDF를 추가할 수 있어야 함
- 파일 리스트에서 순서 변경 가능
- 파일 삭제 가능
- 병합 후 저장 가능
- 총 파일 수 표시

### UX 요구사항
- 드래그 앤 드롭으로 순서 변경
- 순번 자동 표시
- 저장 전 파일명 지정 가능

---

## 9.3 기능 3: PDF 용량 조절

### 목적
제출 제한에 맞춰 PDF 파일 용량을 조절한다.

### 입력
- PDF 파일

### 출력
- 용량 조절된 PDF

### 사용자 노출 옵션
- 최고품질
- 균형
- 소용량
- 목표 용량 입력(고급)

### 선택 이유
비율형 80%, 60%, 40% 방식보다 실제 업무 목적은 "몇 MB 이하로 제출"에 가깝기 때문에 목적형 옵션이 더 적합하다.

### 상세 요구사항
- 기본 옵션은 최고품질
- 변환 시 후처리 또는 별도 화면에서 용량 조절 가능
- 목표 용량은 예: 10MB 이하, 20MB 이하, 30MB 이하처럼 입력 가능
- 목표 용량 달성은 "가능한 한 근접"을 목표로 하며 절대 보장은 하지 않음
- 원본 대비 품질 저하 가능성을 사용자에게 명확히 안내

### UX 요구사항
- 원본 파일 크기 표시
- 결과 파일 크기 표시
- 선택 옵션 표시
- 저장 전 결과 확인

### 파일명 정책

## 13. 기술 스택 확정안

## 13.1 전체 기술 스택 요약
- 데스크톱 프레임워크: Tauri
- 프런트엔드: React
- 언어: TypeScript, Rust
- 스타일링: Tailwind CSS
- 상태 관리: React 기본 상태 + 필요 시 Zustand
- 라우팅: React Router
- 로컬 저장: SQLite + 일부 설정용 JSON
- 빌드/패키징: Tauri Bundler, MSI
- 업데이트: Tauri Updater + GitHub Releases
- 변환 엔진: 로컬 PowerPoint 호출
- PDF 처리: Rust 기반 라이브러리 또는 보조 모듈
- 기본 브라우저 열기: Tauri Shell/Open 연동

## 13.2 왜 Tauri인가
- Windows 데스크톱 앱으로 배포 가능
- 설치 파일(MSI) 생성 가능
- 웹 기술 기반 UI 개발 가능
- Electron보다 상대적으로 가벼운 방향에 적합
- 로컬 파일/프로세스 제어에 유리

## 13.3 왜 React인가
- 자료와 예제가 많음
- 구조화된 컴포넌트 기반 UI 구성에 유리
- 카드형 업무 툴 UI에 적합
- Tauri와 함께 사용하는 사례가 많음

## 13.4 Vue 대신 React를 선택한 이유
- 팀/개발자 입장에서 레퍼런스가 많음
- 유지보수 및 확장 자료가 풍부함
- 향후 컴포넌트 재사용 구조 설계에 유리함

## 13.5 Tailwind CSS 채택 이유
- 가볍고 빠른 UI 개발 가능
- 사내 툴 특성상 정교한 디자인 시스템보다 빠른 구현이 중요
- 카드형/설정형 UI를 빠르게 구성 가능

## 13.6 상태 관리 전략
초기에는 React 기본 상태와 Context를 사용한다.
복잡도가 높아질 경우 아래에 한해 Zustand를 도입한다.

- 작업 큐 상태
- 글로벌 설정 상태
- 변환 진행 상태
- 공통 토스트/알림 상태

## 13.7 로컬 저장 전략
### 설정 데이터
- 기본 저장 경로
- 자동 업데이트 여부
- 결과 폴더 자동 열기 여부
- 최근 UI 설정

저장 방식
- JSON 또는 SQLite 가능
- 단순 설정값은 JSON 권장

### 작업 기록
- 작업 이력은 SQLite 저장 권장
- 이유:
  - 검색/삭제/정렬이 용이
  - 데이터 구조가 안정적
  - 향후 기록 기능 확장에 유리

## 13.8 PDF 처리 계층
PDF 처리 기능은 UI와 분리된 서비스 레이어로 설계한다.

주요 기능
- 병합
- 압축

- 실제 처리 로직은 Rust 서비스 계층에 위임

### 15.1 폴더 구조 설명

#### src/app
전역 앱 구성 요소를 둔다.
- 라우팅
- 전역 provider
- 공통 레이아웃

#### src/pages
화면 단위 엔트리 페이지를 둔다.

#### src/features
기능 중심 모듈 구조를 사용한다.
각 기능은 다음 하위 구조를 기본으로 가진다.
- components
- hooks
- services
- types

이 구조는 기능별 응집도를 높이고 유지보수를 쉽게 한다.

#### src/components
재사용 가능한 공통 컴포넌트를 둔다.

#### src/lib
공통 유틸, 상수, 포맷터, 검증 로직 등 범용 코드를 둔다.

#### src/stores
전역 상태 저장소를 둔다.

#### src/assets
아이콘, 이미지, 브랜딩 리소스를 둔다.

#### src-tauri/src/commands
프런트엔드에서 호출하는 Tauri command 진입점을 둔다.

#### src-tauri/src/services
실제 비즈니스 처리 로직과 외부 프로그램 호출 로직을 둔다.

#### resources
기본 즐겨찾기, 도움말 문서, 브랜딩 리소스를 저장한다.

#### docs
PRD, 기술 명세, 릴리스 노트 등 프로젝트 문서를 관리한다.

#### database
SQLite 마이그레이션과 초기 스키마 관련 파일을 관리한다.

#### .github/workflows
GitHub Actions CI/CD 파이프라인을 둔다.

## 20. 리스크 및 대응

### 리스크 1. 폰트 누락
문서 작성자가 사용한 폰트가 다른 PC에 없을 수 있다.

대응
- 안내 문구 노출
- 최종 PDF 확인 유도

### 리스크 2. PowerPoint 자동화 실패
특정 환경에서 PowerPoint 호출이 실패할 수 있다.

대응
- 상세 오류 메시지
- 실패 로그 저장
- 재시도 안내

### 리스크 3. 목표 용량 정확도 한계
압축 결과가 정확히 특정 MB에 맞지 않을 수 있다.

대응
- "가능한 한 근접" 정책 명시
- 최고품질/균형/소용량 preset 제공

### 리스크 4. 업데이트 접근 이슈
